# 项目版本变更规范

本规范用于 AI 判断是否需要修改版本号、怎么修改版本号，以及公共包消费者如何同步。若用户在当前任务中明确指定版本号，以用户指定为准，并在交付说明里解释原因。

## 基础坐标

Java / Spring / Gradle 新项目默认使用以下基础坐标：

```groovy
group = 'com'
version = '1.0.0'
```

- 已经发布过的项目保持现有 `group` 坐标，不因为补文档或小版本升级强行迁移；修改 `group` 属于发布坐标迁移，必须单独评估 Maven 仓库、消费者依赖和回滚方案。
- 非 Java 项目没有 Gradle `group`；如果存在 `package.json`、小程序上传版本或其它制品版本，递增规则仍按本规范执行。
- 构建制品 `version` 不等同于接口版本、Nacos dataId 版本、流量治理头 `X-Release-Version`、数据库乐观锁 `version` 字段。

## 什么时候改 version

- 修改生产代码、运行时配置、构建脚本、数据库脚本、公共示例，或会改变制品行为的依赖版本时，本批未提交改动只提升一次 `version`。
- 纯 README、AI 编程规范、注释说明、非运行时文档改动，不强制提升制品 `version`。
- 只修改 Nacos 远程配置且没有提交仓库文件时，不修改仓库构建版本，但交付说明必须写清 dataId、group、namespace 和验证结果。
- 同一批改动后续补测试、补文档、修格式，不重复提升版本。

## 怎么改 version

项目内部采用小步补丁递增规则：

- 兼容修复、兼容内部增强、配置治理、测试修复、非破坏性调整：`x.y.z -> x.y.(z + 1)`。
- 当补丁位已经到 `9` 时，下一次升级进位：`x.y.9 -> x.(y + 1).0`。
- 不跳号：例如 `1.4.0` 默认升 `1.4.1`，不能直接升 `1.5.0`，除非用户明确指定或已经批准新的 release line。
- 不兼容 API、数据库、配置、RPC 契约或包坐标变化，才评估 minor / major 级别，并必须列出迁移步骤、消费者影响和回滚方案。

## 公共包和消费者

- `utils` 等公共包改动后，先提升公共包版本并发布到 Maven Local 验证。
- 消费者只引用已经存在的精确版本，不使用动态版本、版本范围或未发布的版本。
- 消费者升级公共包后，必须至少执行编译验证；涉及认证、租户、多数据源、Seata、Dubbo、网关头、序列化或数据库行为时补充测试。
- 从 `main` 分支提交并推送公共包时，必须同步发布远程私有 Maven 制品；消费者仓库再更新依赖版本并验证。

## 交付检查

- 最终说明写明旧版本、新版本、升级原因、是否发布 Maven、消费者是否同步、执行过哪些测试。
- docs-only 任务要明确说明未提升制品版本的原因。
