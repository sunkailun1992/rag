# AI 设计模式规范

本规范约束 AI 在 `rag` 服务中选择、引入和调整设计模式的方式。目标是让报告生成、解析、归档、导出、审核、文件访问、MQ 和通知通道逻辑清晰可扩展。

## 1. 总原则

- 先识别当前语言、框架和模块边界；本项目默认按 Java 17、Spring Boot、MyBatis-Plus、RabbitMQ、Gradle 和 `com:utils` 生态落地。
- 优先沿用当前分层：Controller、BO、Query、VO、Entity、Mapper、Service、ServiceQuery、ServiceResults。
- 设计模式必须服务报告边界：报告归属、访问权限、租户隔离、文件安全、重复生成、MQ 失败处理。
- 不允许为简单 CRUD 硬套 Factory、Manager、Abstract 层或过深继承。
- 公共工具和跨微服务能力优先回到真实同级 `../utils`，不在 `rag` 内复制公共源码。

## 2. 标准参考

- GoF 设计模式：Strategy、Adapter、Factory、Template Method、Observer、State、Builder 等。
- SOLID 原则：判断职责拆分、依赖方向和接口稳定性。
- GRASP 原则：判断报告职责应该归属记录、文件、任务、审核、导出还是通知通道。
- Martin Fowler 企业应用模式：Service Layer、Repository/Mapper、DTO、Transaction Script。
- Spring / RabbitMQ 官方惯例：消息生产、消费、重试、事务边界、监听器和配置分层。

## 3. 本项目推荐模式

### Application Service / Service Layer

- Controller 只处理 HTTP 入参、权限注解和统一响应。
- Service 编排报告记录、报告文件、生成任务、审核状态、MQ 和事务。
- 管理侧、个人侧、异步任务侧和外部回调侧是不同权限场景，不要混用同一个数据权限判断。

### Repository / Mapper

- Mapper 只处理数据库访问。
- 查询条件和排序优先放 ServiceQuery。
- 返回转换和枚举说明优先放 ServiceResults。

### Strategy

适用报告类型、生成器、解析器、导出格式和通知方式。

- 不同报告类型、生成方式、导出格式或通知通道用策略隔离。
- 策略必须明确租户、归属人、可见范围和内容安全边界。
- 不把所有生成、导出和通知分支堆在一个 Service 方法里。

### Observer / Publisher

适用报告生成、审核或归档后需要触发多种后续动作。

- 生成成功、失败、审核通过、归档完成等事件可考虑发布事件或观察者。
- 通知失败不能导致数据库状态不一致，除非业务明确要求事务绑定。
- 事件内容必须脱敏，不能包含密钥、完整请求体或敏感用户资料。

### State

适用报告生命周期和生成任务状态。

- 简单状态用枚举即可。
- 有非法迁移、补偿、重试、撤回、审核等规则时集中状态迁移。
- 状态迁移必须可测试，不能散落在 Controller、消费者和定时任务中。

### Template Method / Pipeline

适用报告生成或导出固定流程。

- 可按校验入参、解析数据源、写生成任务、投递 MQ、更新状态、通知失败处理组织流程。
- 任一关键校验失败应失败关闭，不继续生成或导出。
- 流程稳定后再抽模板或流水线，不为了单一场景提前抽象。

### Adapter

适用对象存储、PDF/Excel 生成器、第三方短信、邮件、微信、App 推送等外部能力。

- 外部能力参数、签名、失败码和重试语义封装在适配器里。
- 上层 Service 只依赖统一报告文件、生成器或通知通道接口。

## 4. 谨慎或禁止使用

- 手写 Singleton：Spring Bean 已管理生命周期。
- Service Locator：优先构造器注入。
- 巨型 Manager：不要把生成、解析、审核、导出、文件访问、MQ、第三方通道全塞进一个类。
- 过深继承：报告类型、生成器或导出格式差异优先用接口和组合表达。
- 全局静态可变状态：报告生成状态和重试上下文必须可追踪。
- 模式先行重构：没有重复实现、稳定扩展点或维护痛点时不改结构。

## 5. 检查清单

- 是否区分管理侧、个人侧、异步任务侧和外部回调侧权限？
- 是否复用了 `utils` 公共能力而非复制公共源码？
- 是否没有绕过认证、租户、数据权限、文件访问安全和统一响应？
- 新模式是否解决真实重复、生成器扩展、导出格式扩展或状态迁移问题？
- 是否存在更简单的函数、枚举、接口或组合方案？
- 是否补充 Controller 请求层、Service 或 MQ 相关测试？
