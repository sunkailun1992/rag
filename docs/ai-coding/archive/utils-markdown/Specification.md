# Java 开发规范压缩版

本文件用于替代历史图片版规范，方便 AI 直接阅读和执行。具体编码时优先结合 `PROJECT_CODING_SPEC.md`、`UTILS_PUBLIC_SPEC.md` 和项目现有代码。

## 命名规范

- 包名全部小写，并与当前微服务已有包结构保持一致。
- 类名使用大驼峰，能直接表达职责，例如 `XxxController`、`XxxServiceImpl`。
- 方法名、参数名、局部变量名使用小驼峰，避免拼音缩写和无意义短名。
- 常量名使用大写下划线，表达稳定业务含义。
- 枚举类建议以 `Enum` 结尾，枚举项使用大写下划线或项目已有风格。
- DTO、BO、VO、Query、Entity、Mapper、Service、Controller 后缀按分层语义使用，不混用。

## 代码结构

- Controller 只处理请求参数、权限注解和统一返回，不堆复杂业务逻辑。
- Service 负责业务编排、事务边界、权限上下文使用、数据转换和异常处理。
- Mapper 只处理数据库访问，普通查询优先使用 MyBatis-Plus 条件构造器。
- Entity 映射数据库表字段，公共字段继承 `EntityBase`。
- 业务枚举放当前模块 `entity.enums`，不要放进公共基类。
- 单个方法只做一类清晰职责，复杂逻辑拆成有业务含义的私有方法。

## 注释规范

项目注释规则以 `PROJECT_CODING_SPEC.md` 的“注释规范”为准。本归档文件只保留通用原则：注释说明业务含义和设计原因，不写无意义翻译。

## 集合与空值

- 对外返回集合时优先返回空集合，不返回 `null`。
- 遍历集合前先明确是否允许为空，避免空指针。
- 判断字符串使用项目已有工具或 `StringUtils`，不要手写脆弱判断。
- 集合转换优先使用清晰的循环或 Stream，避免多层嵌套导致可读性下降。
- Map 只用于明确的动态结构，不作为 Controller 最外层统一响应壳。

## 异常处理

- 业务失败使用项目已有返回码或业务异常，不直接吞异常。
- 捕获异常后必须保留有效上下文，不能空 `catch`。
- 事务方法使用 `@Transactional(rollbackFor = Exception.class)`。
- 需要回滚的业务异常必须能触发事务回滚。
- 对用户暴露的错误信息要稳定、明确，内部细节通过日志定位。

## 并发与幂等

- 先查询后修改必须携带查询得到的数据库 `version`，由 MyBatis-Plus 乐观锁控制并发覆盖。
- 重复提交场景使用项目幂等能力，不靠前端按钮禁用作为唯一保护。
- Redis key 必须包含足够业务维度，避免跨租户、跨用户、跨业务互相影响。
- 分布式锁、幂等锁必须设置过期时间，并在异常路径清理可清理资源。

## 数据库规范

- 表字段命名使用小写下划线。
- 每张业务表按项目规范保留公共字段，例如 `code`、`description`、`version`、`tenant_id`、`is_delete`。
- 状态字段必须有对应枚举，整数状态枚举实现 `IEnum<Integer>`。
- 新增表结构或字段变更必须同步记录 SQL 和模型设计。
- 业务查询不要重复手写 `tenant_id` 和 `is_delete` 条件。
- 不在循环里发起大量单条数据库查询，必要时批量查询后内存组装。

## 接口规范

- Controller 统一返回 `ApiResponse`。
- 成功返回使用 `ApiResponse.success(data)`。
- 需要鉴权的接口必须加 `@PreAuthorize`。
- 权限码使用稳定格式，例如 `module:resource:action`。
- 请求参数要有明确校验，错误时返回稳定错误码。
- 对外 VO 不直接暴露内部敏感字段。

## 安全规范

- 不在日志中输出密码、token、密钥、身份证、手机号完整值等敏感信息。
- 认证用户从 `UserContextHolder` 或 Spring Security 上下文获取，不再使用旧 token header 和 Redis token 用户模式。
- 租户信息从租户上下文或统一 header 解析，不在业务代码里随意覆盖。
- SQL 参数必须经过项目校验能力，不拼接未校验用户输入。
- 跨租户查询必须显式声明并在 `finally` 清理忽略标记。

## 日志规范

- 请求日志记录关键业务维度，例如用户、租户、接口、结果和耗时。
- 异常日志保留定位上下文，但不泄露敏感数据。
- 日志内容要便于排查问题，不写无意义占位文本。
- 高频路径避免输出大量冗余日志。

## AI 生成代码检查

- 是否符合当前微服务包名和模块分层。
- 是否使用 Java 17、Spring Boot 4、`jakarta.*`。
- 是否继承或复用项目公共能力，而不是重复造工具。
- 是否使用 `ApiResponse`、`ReturnCode`、`IEnum`、`EntityBase` 等项目约定。
- 是否补齐类、字段、方法和关键行注释。
- 是否避免新增运行时无关规范资源。
- 是否运行 `./gradlew clean compileJava -x test` 验证。
